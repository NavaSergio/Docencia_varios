Propósito de la sesión

  • Entender los principios básicos para organizar datos en un proyecto de investigación.
  • Crear un plan para recolectar información de manera ordenada y ética.
  • Preparar los elementos necesarios para la Evidencia 4A (base de datos) y presentación final.

Tip

¿Qué vamos a crear hoy?
1) Esquema de organización de datos (¿una tabla o varias?).
2) Diccionario de variables con descripción y transformaciones de cada campo.
3) Pipeline de preparación: limpieza, imputación y agregación.
4) Plan de recolección de datos.
5) Una muestra pequeña de nuestra base de datos lista para análisis.

Conexión con la sesión de pregunta de investigación

  • Pregunta general → define qué fenómeno quieres explicar y el nivel de análisis.
  • Preguntas específicas → delimitan dimensiones observables del problema.
  • Objetivos e hipótesis → indican qué relaciones debes medir o contrastar.
  • Variables e indicadores → se convierten en columnas de tu base de datos.

Important

En esta unidad respondemos: “¿qué estructura de datos necesito para contestar mi pregunta de investigación con evidencia verificable?”

Del hilo conductor a la base para análisis

Si no puedes ubicar una variable en este flujo, todavía no está lista para entrar a tu base.

Pasos para organizar nuestros datos

  1. Partir de la pregunta → ¿Qué debo observar para responder mi pregunta general y específicas?
  2. Diseñar estructura → ¿Una tabla o múltiples? ¿Qué campos necesito?
  3. Definir relaciones → ¿Cómo se conectan los datos entre sí?
  4. Establecer reglas → ¿Qué datos son válidos y cuáles no?
  5. Documentar todo → Crear un diccionario que explique cada variable.
  6. Planear recolección → ¿De dónde y cómo obtendré los datos?
  7. Transformar y preparar → Imputar, agregar y derivar variables según tu pregunta de investigación.
  8. Verificar calidad → ¿Cómo me aseguro de que los datos sean confiables?

De variables a organización de datos

  • Variable dependiente: lo que quiero explicar o predecir.
  • Variables independientes: los factores que pueden influir.
  • Variables derivadas: nuevas variables que calculo a partir de las anteriores.

Ejemplo (estudio de fraude en transacciones):
- Variable dependiente: es_fraude (sí/no).
- Variables independientes: monto, canal, tienda, cliente, fecha_hora.
- Variables derivadas: hora_del_dia, dia_semana, monto_promedio_ultimo_mes.

Grupos de información
- clientes (información personal)
- tiendas (información comercial)
- transacciones (cada compra)

¿Cómo se relacionan?
- Un cliente puede hacer muchas transacciones
- Una tienda puede recibir muchas transacciones
- Cada transacción pertenece a un cliente y una tienda

¿Una tabla o múltiples tablas?

La respuesta depende de tu unidad de observación y de cómo llegaron los datos:

Modelo relacional (múltiples tablas)

  • Datos provenientes de sistemas diferentes
  • Hay relaciones de muchos-a-muchos
  • Ejemplo siguiente: clientes + tiendas + transacciones

Formato plano (una sola tabla)

  • Cada fila es una observación completa
  • Encuestas, sensores o registros de una sola fuente
  • Dataset ya integrado desde el origen
  • La mayoría de repositorios públicos y Kaggle

Important

Independientemente del formato de origen, la mayor parte del análisis estadístico requiere una sola tabla lista para analizar. Si partes de múltiples tablas, eventualmente las integrarás en una para el análisis.

Ejemplo práctico — Estructura en SQL

-- Tabla principal de transacciones
CREATE TABLE transacciones (
  id_transaccion TEXT PRIMARY KEY,
  id_cliente TEXT NOT NULL,
  id_tienda TEXT NOT NULL,
  fecha_hora TIMESTAMP NOT NULL,
  monto DECIMAL(10,2) CHECK (monto >= 0),
  canal TEXT CHECK (canal IN ('pos','web','app','telefono')),
  es_fraude BOOLEAN NOT NULL DEFAULT 0
);

-- Tabla de clientes
CREATE TABLE clientes (
  id_cliente TEXT PRIMARY KEY,
  segmento TEXT,
  fecha_registro DATE
);

-- Tabla de tiendas
CREATE TABLE tiendas (
  id_tienda TEXT PRIMARY KEY,
  categoria TEXT,
  ciudad TEXT
);

Conceptos clave para organizar datos

  • Identificadores únicos: cada fila en una tabla debe tener un código único.
  • Referencias entre tablas: usar códigos para conectar información relacionada.
  • Tipos de datos apropiados: números para cantidades, texto para nombres, fechas para tiempo.
  • Reglas de validación: establecer qué valores son aceptables.

Reglas básicas
- Cada tabla debe tener un identificador único principal.
- Usar códigos simples para conectar tablas (no texto largo).
- Mantener los identificadores estables (no cambiarlos después).

Evaluación del origen de los datos

Antes de analizar, responde estas 6 preguntas de control:

Dimensión Preguntas clave
Procedencia ¿De dónde provienen los datos? ¿Quién los generó? ¿Son propios o de terceros?
Generación/recolección ¿Cómo se recolectaron? ¿Bajo qué condiciones? ¿Existe protocolo?
Estructura y alcance ¿Cuál es la unidad de observación? ¿Qué periodo cubren? ¿Qué cobertura geográfica/poblacional tienen?
Calidad y confiabilidad ¿Se validaron? ¿Hay faltantes, sesgos o errores conocidos? ¿Son consistentes entre periodos/fuentes?
Transformaciones previas ¿Son datos crudos o procesados? ¿Qué agregaciones/imputaciones se aplicaron? ¿Se integraron múltiples fuentes?
Restricciones y uso ¿Hay límites legales, éticos o de licencia? ¿El propósito original es compatible con tu proyecto?

Tip

Versión compacta: Procedencia · Generación · Estructura · Calidad · Transformaciones · Restricciones. Documenta respuestas breves en tu README.md o protocolo.

Important

Si no puedes responder con evidencia documental estas 6 dimensiones, tu base aún no está lista para análisis inferencial.

Control de calidad de los datos

La evaluación del origen responde qué preguntar antes de recolectar; el control de calidad responde cómo verificar lo que ya tienes.

Prevención (al momento de capturar)

  • Usar formularios que validen automáticamente los datos.
  • Crear listas de opciones cerradas cuando sea posible.

Detección de errores (después de capturar)

  • Verificar que no hay duplicados donde no debería haberlos.
  • Revisar que los valores estén en rangos razonables.
  • Comprobar que las fechas sean lógicas.

Documentación

  • Registrar de dónde viene cada dato y cuándo se capturó.
  • Mantener los datos originales sin modificar en una carpeta separada.

Imputación: qué hacer con datos faltantes

Identificar el mecanismo de ausencia es el primer paso antes de elegir cualquier método (Little & Rubin, 2002; Rubin, 1976).

Métodos de imputación comunes

Método Cuándo usarlo Ventaja Limitación
Eliminar filas Pocos faltantes (<5%), MCAR Simple Pierde información
Media / mediana Numéricas, MCAR Rápido Subestima la varianza
Moda Categóricas Fácil Sesgo si hay dominancia
Valor anterior (fill) Series de tiempo Respeta la tendencia Solo datos ordenados
Regresión MAR, variables correlacionadas Preciso Requiere más trabajo
Imputación múltiple Análisis formales Estadísticamente robusto Complejo de implementar

Warning

Documenta siempre qué método usaste y en qué variables. El método elegido debe justificarse en tu sección de metodología.

Para imputación múltiple se recomienda consultar Buuren (2018) (disponible en línea), Sterne et al. (2009) y la implementación en R de Buuren & Groothuis-Oudshoorn (2011). Schafer & Graham (2002) ofrece una revisión metodológica accesible.

Consideraciones éticas y de privacidad

  • Minimiza la recolección: solo recolecta los datos que realmente necesitas.
  • Protege la identidad: usa códigos en lugar de nombres cuando sea posible.
  • Controla el acceso: define quién puede ver qué información.
  • Datos sintéticos: si necesitas compartir datos, considera generar versiones artificiales que mantengan las características importantes pero no revelen información personal.

¿Qué son los datos sintéticos?

Los datos sintéticos son conjuntos de datos artificiales que imitan las características estadísticas de datos reales sin contener información real de personas específicas.

¿Cuándo usarlos?

  • Para compartir datos con colaboradores
  • Para publicar datasets de investigación
  • Para entrenar modelos sin exponer datos sensibles
  • Para crear ejemplos educativos

Ventajas

  • Protegen la privacidad individual
  • Mantienen patrones estadísticos útiles
  • Permiten reproducibilidad
  • Eliminan restricciones legales de uso

Lista de verificación para la Evidencia 4A

Actividad

  1. Decide si tu proyecto requiere una tabla o varias, y justifica la respuesta.
  2. Construye una mini matriz de coherencia: pregunta específica → objetivo → hipótesis → variable/métrica.
  3. Dibuja tu pipeline: ¿qué pasos debes aplicar desde los datos crudos hasta el dataset listo para análisis?
  4. Define los campos principales (8-10 variables clave) e indica si alguna requiere imputación, agregación o derivación.
  5. Aplica el checklist de evaluación del origen de datos y registra una evidencia por cada dimensión.
  6. Escribe 3 posibles problemas de calidad y cómo los resolverías (duplicados, valores faltantes, valores fuera de rango).
  7. Describe tu plan de recolección: fuente, granularidad temporal/espacial y frecuencia.

Puntos clave para recordar

  • Planifica antes de recolectar: diseña tu estructura de datos antes de empezar a capturar información.
  • Documenta todo: mantén un diccionario actualizado de qué significa cada variable y qué transformaciones se le aplicaron.
  • Prepara, no solo recolecta: casi siempre necesitarás limpiar, imputar o agregar antes de analizar.
  • Elige la granularidad correcta: la unidad de análisis (fila) debe corresponder con tu pregunta de investigación.
  • Considera la ética: protege la privacidad y recolecta solo lo necesario.

La calidad de tu análisis depende directamente de la calidad y preparación de tus datos.
